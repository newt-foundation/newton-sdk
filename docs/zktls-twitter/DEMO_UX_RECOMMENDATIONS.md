# Minimal Demo UX Recommendations

Goal: make Newton zkTLS understandable to a first-time external user in under five minutes.

Summary: the demo UX should make four steps clear on screen: proof generation, CID handoff, Newton submission, and verification result.

## Recommended demo screen flow

1. **Connect / choose account**
   - Show wallet or app identity.
   - Keep chain and gateway status visible.

2. **Choose Twitter/X proof goal**
   - Example: “Prove I have at least 1,000 followers.”
   - Show editable threshold.

3. **Generate or paste TLSNotary proof CID**
   - For live demo: invoke proof-generation UX.
   - For dry demo: paste a known fixture CID and label it “fixture”.

4. **Preview Newton task payload**
   - Show `proof_cid`, `policy_client`, `wasm_args` summary.
   - Hide or collapse raw calldata by default.

5. **Submit to Gateway**
   - Show request status: `queued`, `operator verifying`, `policy evaluated`, `attestation ready`.

6. **Show result and explain trust boundary**
   - Result: allowed / denied.
   - Explain that operators verify TLSNotary proof bytes, not a screenshot or self-reported follower count.

## Copy guidelines

Use plain language:

- “Proof CID” → “IPFS link to your TLSNotary proof”
- “zkTLS” → “cryptographic proof of a real HTTPS response”
- “Operator verification” → “Newton operators independently check the proof”

Plain-language wording:

- zkTLS: “cryptographic proof of a real HTTPS response”
- proof CID: “IPFS address for the stored proof file”
- operator: “verification node that checks the proof”

## Error UX checklist

For every failed demo submission, show:

- What failed: gateway, IPFS fetch, TLSNotary verification, or policy rule.
- What the user can do next.
- A copy button for the exact failing command or request ID.

## Minimum viable demo assets

- One dry-run payload generated by `scripts/zktls-twitter-sdk-demo.sh`
- One policy explainer snippet for the operator-side `tlsn_twitter_followers.rego` policy in `newton-prover-avs`
- One status timeline with four states:
  1. Payload ready
  2. Proof fetched
  3. Proof verified
  4. Policy decision returned

## Avoid in the first demo

- Do not start with raw Rego syntax.
- Do not expose all Docker logs unless troubleshooting.
- Do not call placeholder CIDs “valid proofs”.
- Do not imply Twitter/X endorses or participates in the flow.
